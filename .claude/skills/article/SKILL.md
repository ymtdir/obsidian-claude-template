技術記事（Zenn / Qiita 等）を執筆し、`50_ARTICLES/` に保存する。10_WIKI を参照してファクトチェックし、閲覧数を稼ぐ構成（タイトル・TL;DR・結論ファースト）に沿って書く。

引数: `$ARGUMENTS`（先頭の `zenn` / `qiita` でプラットフォームを指定。省略時は `zenn`）

## 手順

### 1. トピックの確定と調査

- 引数からプラットフォームとトピックを特定する
- `10_WIKI/` を Grep で関連ノート検索（類義語・関連技術も含む）
- 既存 WIKI ノートの内容を Read で読み、記事の土台にする
- 不足する部分は WebSearch で補完し、**一次情報**（公式ドキュメント・RFC・論文等）を優先する
- Claude の知識は補助にとどめ、推測で事実を書かない

### 2. ファクトチェック

- 記事で主張する**技術事実・バージョン情報・API 仕様・数値**は必ず一次情報で裏取りする
- `10_WIKI/` の既存記述と食い違いがあれば、どちらが正しいか WebSearch で確認してから書く
- 裏取りできなかった主張は記事に含めない（または「要検証」と明示）

### 3. 構成の設計（閲覧数を稼ぐ）

記事は次の構造で組む:

- **タイトル**: 具体 + ベネフィット（例: 「Rust の所有権を図解で 5 分で理解する」）。釣りすぎず、検索キーワードを自然に含む
- **TL;DR / 結論ファースト**: 冒頭で結論・対象読者・読後に何が得られるかを提示
- **課題・動機**: なぜこのトピックが重要か（「こういう場面で困る → この記事を読めば解決」）
- **本文**: 見出しを `##` から階層的に。主要セクション（`##`）内で「**概念の整理 → コード例 → 落とし穴**」を `###` で並列配置する。題材が複数ある場合は主要セクションを題材ごとに分け、各セクション内で繰り返す
- **コード例**: 動く最小構成を示す。コメントで要点を添える
- **まとめ + 次のアクション**: 要点 3 つ以内 + 次に読むべきリソース（公式ドキュメント等）

### 4. `template.md` に沿って執筆

- `.claude/skills/article/template.md` を読み、その構造で本文を書く
- プラットフォーム別の記法差を適用:
  - **Zenn**: `:::message`（補足）/ `:::message alert`（警告）/ `:::details`（折りたたみ）
  - **Qiita**: `:::note info` / `:::note warn` / `:::note alert`
- コードブロックは言語指定を忘れない（`\`\`\`rust`、`\`\`\`ts` 等）
- 画像参照は `99_ASSETS/` に置き、相対パスで参照

### 5. `50_ARTICLES/` に保存

- ファイル名は記事タイトルを端的に表す名前（日本語 OK、`[[wikilink]]` 表記と同一）
- プラットフォーム別にサブディレクトリを切る: `50_ARTICLES/zenn/` / `50_ARTICLES/qiita/`
- **保存前に `50_ARTICLES/<platform>/` を Glob で事前チェック** し、同名ファイルの衝突を検知する
- 衝突時は **既存ファイルを Read して切り口を把握** し、重複を避ける切り口で新記事を書く
- ファイル名は **括弧補足で差別化** する（例: `Rust所有権（入門）.md`、`GraphQLスキーマ設計（N+1問題編）.md`）
- **frontmatter は書かない**（プラットフォーム側の CLI / Web UI で後から付ける想定。本 skill は記事本文のみを担う）
- 見出しは `##` から使い、`# タイトル` は書かない（ファイル名がタイトルを担う）
- `[[wikilink]]` は張らない（外部公開先でリンクが壊れるため）

### 6. レポート

- 作成した記事のファイルパス、タイトル、対象プラットフォーム、記事の概要を提示する
- ファクトチェックで `10_WIKI/` を参照したノート、WebSearch で裏取りした一次情報も明示する
